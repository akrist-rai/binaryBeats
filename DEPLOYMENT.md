# Deployment

This project has two independently-deployed pieces:

| Piece | What it is | Where it goes |
|---|---|---|
| **Frontend** | Vite/React static build (`src/`) | Vercel |
| **Backend** | Koa API (`server/`) — Postgres (Neon) + a local C++ judge | Render (default), or any Docker host |

They're separate deployments on purpose. The sections below cover the default
setup (Vercel + Render) and how to swap the backend host later without
touching frontend code.

## Why the backend can't go on Vercel

Vercel functions are stateless and ephemeral — no persistent process, no
guaranteed filesystem between invocations, and no `g++`. This backend needs
all three:

- **The judge** (`server/src/judge/executor.ts`) shells out to `g++` to
  compile submitted C++, then runs the binary against test input read from a
  temp directory on disk.
- **`app.listen()`** is a long-running Koa process, not a request handler.
- **The session poller** (`server/src/sessionPoller.ts`) runs on an interval
  in the background, independent of any incoming request.

So the backend needs a real container that stays running. Render, Railway,
Fly.io, and a plain VPS all work identically here because deployment is
Docker-based (`server/Dockerfile`) — switching between them is mostly "point
a different platform at the same Dockerfile."

## 1. Frontend → Vercel

1. Push this repo to GitHub/GitLab/Bitbucket and import it in the [Vercel
   dashboard](https://vercel.com/new), or run `npx vercel` from the repo
   root.
2. Vercel auto-detects Vite; `vercel.json` at the repo root pins the build
   (`npm run build`, output `dist`) explicitly so this doesn't depend on
   detection.
3. `.vercelignore` excludes `server/` (its secrets and its multi-GB
   `node_modules`/data have no reason to leave your machine) and the design
   prototype files (`design/`, `DESIGN.md`, `extra/`, `login.html`,
   `access.html`), which aren't part of the Vite build.
4. In the Vercel project's **Settings → Environment Variables**, add:

   | Key | Value |
   |---|---|
   | `VITE_API_URL` | The backend's public URL, e.g. `https://binary-beats-api.onrender.com` (no trailing slash) |

   Every frontend API client (`src/lib/apiBase.ts`) reads this at build time.
   Leaving it unset makes all API calls relative (`/api/...`), which only
   works if frontend and backend share a domain — not the case with Vercel +
   Render, so **set it** before the first deploy.
5. Deploy. Redeploy whenever you change `VITE_API_URL` (Vite bakes env vars
   into the build, so it won't pick up a change until the next build).

## 2. Backend → Render (default)

1. In the [Render dashboard](https://dashboard.render.com), **New → Blueprint**,
   point it at this repo. Render reads `render.yaml` at the repo root and
   provisions a Docker web service from `server/Dockerfile`.
   (No blueprint? **New → Web Service** → same repo → environment **Docker**
   → Dockerfile path `server/Dockerfile`, Docker context `server` — same
   result, done by hand.)
2. Set the environment variables Render's dashboard prompts for (from
   `render.yaml`'s `sync: false` entries):

   | Key | Required | Where to get it |
   |---|---|---|
   | `DATABASE_URL` | Yes | [neon.tech](https://neon.tech) → your project → Connection Details (problem/test dataset). The server throws on startup without it. |
   | `AUTH_DATABASE_URL` | Yes, for auth | A **separate** Neon project's connection string (holds only the `users` table — kept apart from the dataset DB since that one's already near its free-tier storage cap). Without it, `/api/auth/*` responds `503 AUTH_NOT_CONFIGURED` instead of crashing the rest of the API — a common half-deployed state, not a startup failure. |
   | `SESSION_SECRET` | Yes | Auto-generated by the blueprint (`generateValue: true`). Manually: `openssl rand -hex 32`. The server throws on startup without it. |
   | `GOOGLE_CLIENT_ID` | No | [Google Cloud Console](https://console.cloud.google.com/apis/credentials) → OAuth client ID → Web application. Leave unset to disable the Google sign-in button. |

   `PORT` doesn't need setting — Render injects it and `server/src/index.ts`
   already reads `process.env.PORT`.
3. Deploy. Render builds the image (see `server/Dockerfile` — it installs
   `g++` and `util-linux`/`prlimit` for the judge, builds TypeScript in a
   build stage, then ships only the compiled `dist/` in the runtime image)
   and health-checks `GET /api/health`.
4. Copy the resulting URL (e.g. `https://binary-beats-api.onrender.com`) into
   Vercel's `VITE_API_URL` (step 1.4 above) and redeploy the frontend.

### Database

The problem dataset lives in Neon Postgres, not on the server's filesystem —
`server/data/` (gitignored, multi-GB) is only the local ingest source used
once via `npm --prefix server run ingest` to populate a fresh Neon database.
You don't need it present to deploy or run the server; you only need it again
if you're standing up a brand new database from scratch.

### Cross-domain cookies

`server/src/auth.ts`'s session cookie and `server/src/index.ts`'s CORS config
already account for frontend and backend being on different domains
(`sameSite: "none"` + `secure: true` once the request arrives over HTTPS,
`app.proxy = true` so that's detected correctly behind Render's reverse
proxy). If you wire up `login.html`/`access.html` to call `/api/auth/*`,
make sure those `fetch()` calls pass `credentials: "include"` — omitted by
default, and without it the browser won't send or store the session cookie
cross-site.

## 3. Switching the backend host later

Nothing on the frontend hardcodes Render. To move the backend:

1. Deploy `server/` (same `Dockerfile`) to the new platform — see below for
   platform-specific notes.
2. Set the same env vars (`DATABASE_URL`, `AUTH_DATABASE_URL`, `SESSION_SECRET`,
   `GOOGLE_CLIENT_ID`) there.
3. Update `VITE_API_URL` in Vercel to the new backend URL and redeploy the
   frontend.
4. Decommission the old backend once you've confirmed the new one works.

### Railway

1. New Project → Deploy from GitHub repo → this repo.
2. Settings → set the root/build context to `server` (or add a
   `railway.json`/service setting pointing `dockerfilePath` at
   `server/Dockerfile` — Railway builds from `server/Dockerfile` directly
   given a Dockerfile in the repo).
3. Add the same env vars under Variables.
4. Railway assigns a public URL and sets `PORT` automatically, same as Render.

### Fly.io

1. `fly launch` from the `server/` directory (it detects `Dockerfile`
   automatically) — decline the offer to provision a Postgres database, since
   you're using Neon.
2. `fly secrets set DATABASE_URL=... AUTH_DATABASE_URL=... SESSION_SECRET=... GOOGLE_CLIENT_ID=...`
3. `fly deploy`. Fly terminates TLS at its edge the same way Render does, so
   `app.proxy = true` still applies.

### Plain VPS (Docker)

1. `docker build -t binary-beats-api ./server`
2. `docker run -d --restart unless-stopped -p 4000:4000 -e DATABASE_URL=... -e AUTH_DATABASE_URL=... -e SESSION_SECRET=... -e GOOGLE_CLIENT_ID=... binary-beats-api`
3. Put a reverse proxy (Caddy/nginx) in front for TLS — that's what makes
   `ctx.secure` true and the cross-domain cookie logic above kick in. Without
   TLS termination, sessions fall back to `sameSite: "lax"`, which won't work
   if the frontend is on a different domain.

### Plain VPS (no Docker)

`server/Dockerfile` is the source of truth for what the runtime environment
needs (Node 20, `g++`, ideally `util-linux` for `prlimit`). Install those
system-wide, then:

```
cd server
npm ci
npm run build
DATABASE_URL=... AUTH_DATABASE_URL=... SESSION_SECRET=... npm start
```

Run it under a process manager (systemd, pm2) so it restarts on crash and on
reboot — plain `npm start` in a terminal won't survive either.
